# 🧬 Skill Evolution Protocol (P2.4)
> Правило автоматической эволюции скиллов агентов.
> **ОБЯЗАТЕЛЬНО** к исполнению при обнаружении нового скилла или паттерна.

## Триггеры активации
Это правило активируется когда:
1. Пользователь приносит **новую технологию / инструмент / паттерн**
2. Обнаружен **новый ECC-скилл** в `.ecc-library/` (после `git pull`)
3. Пользователь просит **обновить/прокачать** агента
4. В ходе работы найден **лучший подход**, заменяющий текущий
5. Обнаружен **"мёртвый" MD-файл** — ценный контент в неактивном месте

## Протокол обновления (5 шагов)

### Шаг 1: Идентификация владельца
Определи, какому агенту принадлежит новый скилл:

| Домен | Агент(ы) |
|-------|----------|
| Деплой, CI/CD, Docker, PM2 | Кулибин, Ботмэн |
| Безопасность | Кулибин |
| LLM, AI-оптимизация | Кулибин, Игорёк |
| Фронтенд, UI, CSS | Артемий, Рембрандт |
| Контент, тексты | Шекспир, Маркетолог |
| Исследования, аналитика, GEO | Шерл, Маркетолог |
| Дизайн, визуал, 3D | Рембрандт, Артемий |
| Оркестрация, архитектура, ADR | Игорёк |
| Telegram-боты, автоматизации | Ботмэн |
| Битрикс24, CRM | Кулибин, Ботмэн |
| СВО, ветераны | svo-veteran-support (проектный) |

### Шаг 2: Сравнение с существующим
Прочитай SKILL.md агента-владельца и ответь:
1. Есть ли уже аналогичная секция?
   - Да → Шаг 3 (сравнение)
   - Нет → Шаг 4 (добавление)

### Шаг 3: Сравнение версий (REPLACE / MERGE / SKIP)
| Критерий | Вес | Вопрос |
|----------|-----|--------|
| **Актуальность** | 40% | Содержит ли новый более свежие паттерны/API/версии? |
| **Полнота** | 30% | Покрывает ли новый больше use-cases? |
| **Практичность** | 20% | Есть ли готовые примеры кода, чеклисты? |
| **Наш контекст** | 10% | Совместим ли с нашим стеком (Astro, Python, PM2)? |

**Решение:**
- Score >= 60% → **REPLACE** (удалить старую секцию, вставить новую)
- Score 40-60% → **MERGE** (дополнить лучшими частями нового)
- Score < 40% → **SKIP**

### Шаг 4: Установка / Замена
Формат секции (обязательно!):
```markdown
## 🏷 Название Скилла (из ECC Library)
> Источник: .ecc-library/rules/НАЗВАНИЕ
> Дата: YYYY-MM-DD

### Содержание...
```
**Место вставки:** ПЕРЕД секцией `## Constraints`

### Шаг 5: Обновление реестра
После установки обновить `freelance-agent/.agent/skills-registry.json`:
```json
{
  "name": "НАЗВАНИЕ",
  "source": ".ecc-library/rules/НАЗВАНИЕ.md",
  "installed_date": "YYYY-MM-DD",
  "target_skill": "agent-name/SKILL.md",
  "status": "active"
}
```

## Правила замены

### ✅ Заменять когда:
- API/версия в старом скилле устарела (Haiku 3.5 → Haiku 4.5)
- Появился ECC v2 скилла
- Наш кастомный паттерн уступает ECC по полноте

### ❌ НЕ заменять когда:
- Наш кастомный контент содержит **уникальный контекст проекта** (Анжелочка, Битрикс)
- ECC скилл для другого стека без адаптации
- Новый скилл дублирует уже установленный без улучшений

### 🔀 Мержить когда:
- Новый покрывает 50-70% старого, но старый имеет уникальные секции
- Оба содержат ценные, но разные чеклисты

## Антипаттерн: "Мёртвый контент"
Если найден ценный MD-файл вне цепочек использования:
```
Найти → Прочитать → Определить владельца → Интегрировать в SKILL.md
                                               ↓
                                     НЕ оставлять как есть!
```
Примеры мёртвого контента (найдены в аудите 2026-04-30):
- `src/agents/specialists/REMBRANDT_3D_SKILL.md` → интегрирован в rembrandt
- `prompts/library/marketer.md` → интегрирован в marketer-strategist
- `templates/ai-agent-implementation/BITRIX_AGENT_CHECKLIST.md` → стал bitrix-integration/SKILL.md
- `svo-start/skills/` → стали svo-veteran-support/SKILL.md
