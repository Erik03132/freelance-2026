```
❌ task-prioritizer НЕ НАХОДИТ ошибки в коде
❌ task-prioritizer НЕ ДАЁТ РЕКОМЕНДАЦИИ по улучшению
❌ task-prioritizer НЕ СКАНИРУЕТ КОД
❌ task-prioritizer НЕ ПРОВЕРЯЕТ КАЧЕСТВО КОДА
```

```
❌ Врач, который осматривает пациента и говорит:
   "У вас есть проблема с сердцем, нужно лечение"

❌ Механик, который смотрит на машину и говорит:

❌ Аудитор, который проверяет компанию и говорит:
```

```
✅ Секретарь, который смотрит на кучу писем и говорит:

    1. Срочный заказ

    3. Обычный заказ"
```

## КОНКРЕТНЫЙ ПРИМЕР

### Сценарий: У тебя есть GitHub репо с 50 issues

**ТЫ ДАЁШЬ:**
```
Ссылка на репо: https://github.com/myname/myproject
```

**task-prioritizer ДЕЛАЕТ:**
```
1. Получает список всех 50 issues из GitHub
2. Смотрит на каждый issue:
   - Какие labels (теги) у него?
   - Когда он был создан?
   - Сколько комментариев?
3. Считает скор для каждого issue
4. Ранжирует их по скору
5. Показывает результат:

🔴 КРИТИЧНО (100/100)
1. GH-123 — Production database down
   └─ production_critical (40) + performance_issue (30) + database S-tier (30) + ...

🔴 КРИТИЧНО (95/100)
2. GH-456 — Security vulnerability in auth
   └─ security_issue (35) + auth S-tier (30) + ...

🟠 ВЫСОКИЙ (60/100)
3. GH-789 — API endpoint returns 500 error
   └─ api_breaking (28) + api_gateway (30) + ...

🟢 НИЗКИЙ (20/100)
4. GH-101 — Add dark mode feature
   └─ feature_request (15) + frontend (8)

...и ещё 46 issues, отсортированных по приоритету
```

**task-prioritizer НЕ ДЕЛАЕТ:**
```
❌ НЕ смотрит в код
❌ НЕ находит ошибки
❌ НЕ создаёт новые issues
❌ НЕ даёт рекомендации по улучшению
❌ НЕ проверяет качество кода
```

**Когда использовать:**

- Интеграция в workflow

## ЧТО НУЖНО СДЕЛАТЬ ВРУЧНУЮ?

task-prioritizer НЕ ДЕЛАЕТ ничего автоматически. **ТЫ ДОЛЖЕН:**

### Шаг 1: Создавать issues вручную

```
Когда ты находишь проблему, ты САМИ создаёшь issue:

Title: "Fix: Database connection timeout"
Labels: bug, production, critical
Description: "When users connect from mobile, database times out..."
```

**task-prioritizer НЕ создаёт issues.**

---

### Шаг 2: Добавлять labels вручную

```
Когда создаёшь issue, ты САМИ добавляешь labels:

Labels: bug, production, critical
```

**task-prioritizer НЕ добавляет labels.**
**task-prioritizer ИСПОЛЬЗУЕТ существующие labels.**

---

### Шаг 3: Решать issues вручную

```
task-prioritizer ПОКАЗЫВАЕТ приоритет:
"Делай GH-123 в первую очередь"

Но ты САМИ решаешь issue:
1. Открываешь код
2. Находишь проблему
3. Исправляешь
4. Делаешь commit
5. Закрываешь issue
```

**task-prioritizer НЕ решает issues.**

## ЧАСТЫЕ ВОПРОСЫ

### Q: Это найдёт все баги в моём коде?

**A:** НЕТ. task-prioritizer НЕ сканирует код.

Это только ранжирует СУЩЕСТВУЮЩИЕ issues.

Если у тебя нет issues в GitHub, task-prioritizer ничего не найдёт.

---

### Q: Это даст рекомендации по улучшению проекта?

**A:** НЕТ. task-prioritizer НЕ даёт рекомендации.

Это только показывает, какие issues решать в первую очередь.

---

### Q: Это будет сканировать мой проект каждый день?

**A:** ТОЛЬКО если ты это настроишь.

По умолчанию: разовая задача (запускаешь вручную)

Если хочешь автоматизацию: добавь в cron

```
# Каждый день в 9:00
0 9 * * * python3 github_issues_example.py | post_to_slack
```

---

### Q: Это найдёт ошибки в коде?

**A:** НЕТ. Используй для этого:

```
✅ ESLint (для JavaScript)
✅ Pylint (для Python)
✅ RuboCop (для Ruby)
✅ Sonar (для анализа качества кода)
✅ Sentry (для ошибок в production)
```

task-prioritizer работает с issues, а не с кодом.

---

### Q: Это создаст issues автоматически?

**A:** НЕТ. task-prioritizer НЕ создаёт issues.

Ты должен создавать issues вручную или использовать другие инструменты:

```
✅ Sentry (автоматически создаёт issues для ошибок)
✅ CodeClimate (автоматически создаёт issues для качества кода)
✅ Dependabot (автоматически создаёт issues для обновления зависимостей)
```

## КОГДА ИСПОЛЬЗОВАТЬ?

### Используй, если:

```
✅ У тебя много issues в GitHub
✅ Непонятно, какой решать в первую очередь
✅ Нужно быстро понять приоритеты
✅ Хочешь ежедневный отчёт о приоритетах
```

### НЕ используй, если:

```
❌ У тебя нет issues в GitHub
❌ Ты ищешь инструмент для поиска ошибок в коде
❌ Ты ищешь инструмент для аудита проекта
❌ Ты ищешь инструмент для рекомендаций по улучшению
```

Для этого используй:
- **Sonar** (аудит качества кода)
- **Sentry** (отслеживание ошибок)
- **Dependabot** (обновление зависимостей)
- **CodeClimate** (анализ качества)

## ЗАКЛЮЧЕНИЕ

**task-prioritizer = это помощник для приоритизации**

Не это:
```
❌ Аудит проекта
❌ Поиск ошибок
❌ Рекомендации
❌ Сканирование кода
```

Это это:
```
✅ Ранжирование issues
✅ Показ приоритетов
✅ Помощь в выборе, что делать в первую очередь
```

**Используй task-prioritizer для приоритизации, а другие инструменты для аудита и поиска ошибок.**
