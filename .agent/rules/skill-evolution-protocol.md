# 🧬 Skill Evolution Protocol
> Правило автоматической эволюции скиллов агентов Antigravity.
> **ОБЯЗАТЕЛЬНО** к исполнению при обнаружении нового скилла.

## Триггеры активации
Это правило активируется когда:
1. Пользователь приносит **новую технологию/инструмент/паттерн**
2. Обнаружен **новый ECC-скилл** в библиотеке (после `git pull`)
3. Пользователь просит **обновить/прокачать** агента
4. В ходе работы найден **лучший подход**, заменяющий текущий

## Протокол обновления (5 шагов)

### Шаг 1: Идентификация владельца
Определи, какому агенту принадлежит новый скилл по `routing_rules` из `skills-registry.json`:

| Домен | Агент(ы) |
|-------|----------|
| Деплой, CI/CD, Docker | Кулибин, Ботмэн |
| Безопасность | Кулибин, Ботмэн |
| LLM оптимизация | Кулибин, Ботмэн |
| Фронтенд, UI | Артемий |
| Контент, тексты | Шекспир, Маркетолог |
| Исследования, аналитика | Шерл, Маркетолог |
| Дизайн, визуал | Рембрандт, Артемий |
| Оркестрация, архитектура | Игорёк |
| Боты, автоматизации | Ботмэн |

### Шаг 2: Сравнение с существующим
Прочитай SKILL.md агента-владельца и ответь на вопросы:

1. **Есть ли уже аналогичная секция?**
   - Да → переходи к Шаг 3 (сравнение)
   - Нет → переходи к Шаг 4 (добавление)

2. **Какой section_marker** у существующей секции?
   - Найди `## 🏷 Название (из ECC Library)` маркер
   - Определи начало и конец секции (до следующего `## ` или `---`)

### Шаг 3: Сравнение версий (REPLACE vs KEEP)
Сравни новый скилл с установленным по критериям:

| Критерий | Вес | Вопрос |
|----------|-----|--------|
| **Актуальность** | 40% | Содержит ли новый более свежие паттерны/API/версии? |
| **Полнота** | 30% | Покрывает ли новый больше use-cases? |
| **Практичность** | 20% | Есть ли готовые примеры кода, чеклисты? |
| **Наш контекст** | 10% | Совместим ли с нашим стеком (Astro, Python, PM2)? |

**Решение:**
- Score >= 60% в пользу нового → **REPLACE** (удалить старую секцию, вставить новую)
- Score < 60% → **MERGE** (дополнить существующую лучшими частями нового)
- Новый хуже → **SKIP** (не трогать)

### Шаг 4: Установка / Замена
При добавлении нового или замене старого:

1. **Формат секции** (обязательно!):
```markdown
## 🏷 Название Скилла (из ECC Library)
> Источник: everything-claude-code/skills/НАЗВАНИЕ
> Дата установки: YYYY-MM-DD

### Содержание...
```

2. **Место вставки:** ПЕРЕД секцией `## Constraints` (всегда в конце)

3. **Адаптация:** Переведи ключевые паттерны на русский, адаптируй примеры под наш стек

### Шаг 5: Обновление реестра
После установки обязательно обнови `skills-registry.json`:

```json
{
  "name": "НАЗВАНИЕ-СКИЛЛА",
  "source": ".ecc-library/skills/НАЗВАНИЕ/SKILL.md",
  "installed_date": "YYYY-MM-DD",
  "section_marker": "## 🏷 Название (из ECC Library)",
  "status": "active",
  "replaced": "старый-скилл-если-был"   // Только если заменял
}
```

## Правила замены

### ✅ Заменять когда:
- API/версия в старом скилле устарела (пример: Haiku 3.5 → Haiku 4.5)
- Появился ECC v2 скилла (пример: continuous-learning → continuous-learning-v2)
- Наш кастомный паттерн уступает ECC по полноте

### ❌ НЕ заменять когда:
- Наш кастомный контент содержит **уникальный контекст проекта** (Анжелочка, Битрикс)
- ECC скилл для другого стека (Rails, Spring) без адаптации
- Новый скилл дублирует уже установленный без улучшений

### 🔀 Мержить когда:
- Новый покрывает 50-70% старого, но старый имеет уникальные секции
- Оба содержат ценные, но разные чеклисты
- Новый добавляет паттерны, но не заменяет архитектуру

## Автоматическая проверка
Запуск сканера:
```bash
bash ~/.gemini/antigravity/skills/skill-evolution.sh
```

## Пример эволюции
```
БЫЛО: cost-aware-llm-pipeline (Haiku 3.5, $0.25/1M)
СТАЛО: cost-aware-llm-pipeline v2 (Haiku 4.5, $0.80/1M, + prompt caching)

Решение: REPLACE
Причина: Цены устарели, добавлен prompt caching
Действие: Удалить секцию "## 💰 Cost-Aware LLM Pipeline"
          Вставить новую с актуальными ценами
          Обновить registry: replaced = "cost-aware-llm-pipeline"
```
