# 📜 ПРОТОКОЛ ХРОНИКИ ДНЯ (Chronicle Protocol)

## ОБЯЗАТЕЛЬНОЕ ПРАВИЛО ДЛЯ ВСЕХ АГЕНТОВ

### Что это
Хроника дня — единый файл `~/freelance-2026/chronicles/chronicle_YYYY-MM-DD.md`, в который **каждая сессия** дописывает свои действия в реальном времени. Файл переживает падения сессий.

### Когда записывать

**Принцип: ПАЧКОЙ, не на каждый чих.** Экономим токены.

Агент ОБЯЗАН делать запись в хронику:

1. **При старте сессии** — `chronicle.sh --init "session-id"`
2. **Раз в ~15 минут** — накопить 3-5 действий и записать одним вызовом `chronicle.sh`
3. **При ключевых вехах** (не ждать 15 мин):
   - Деплой на VPS
   - Публикация контента (VK, TG)
   - Критический баг / решение
4. **При завершении сессии** — итоговая запись перед finish-day

**НЕ записывать:** каждое чтение файла, каждый grep, промежуточные шаги.

### Как записывать

```bash
# Инициализация сессии (при буте)
bash ~/freelance-2026/tools/chronicle.sh --init "conversation-id"

# Запись действия
bash ~/freelance-2026/tools/chronicle.sh "✅ Описание того, что сделано"

# Проверка статуса
bash ~/freelance-2026/tools/chronicle.sh --status
```

### Формат записи

```
- **HH:MM** — ✅ Что сделано (деталь)
- **HH:MM** — 🔧 Создан файл X — назначение
- **HH:MM** — 🚀 Деплой Y на VPS
- **HH:MM** — ❌ Проблема Z — как решили
- **HH:MM** — 📊 Метрика: было X, стало Y
```

### Где хранится

```
~/freelance-2026/chronicles/
├── chronicle_2026-05-01.md   ← сегодня
├── chronicle_2026-04-30.md
├── chronicle_2026-04-29.md
└── ...
```

### Интеграция с finish-day

При `finish-day` хроника НЕ удаляется — она дополняет report-day:
- `chronicle_*.md` — хронологический лог (что, когда, в какой сессии)
- `report-day_*.md` — аналитический отчёт (итоги, метрики, следующие шаги)

### Почему это важно

- Сессия может оборваться без предупреждения → хроника уже записана
- Несколько сессий за день → все пишут в один файл
- Следующая сессия видит, что делали предыдущие
- При буте GBP → читать хронику дня = знать контекст
