# Управление документами и SSoT (для всех проектов)

## Single Source of Truth (SSoT)
```
Каждой теме — ОДИН файл.
Нет файла по теме → создать.
Есть файл по теме → ДОПОЛНИТЬ, не создавать новый.
```

## Перед созданием нового файла — спроси
1. Существует ли уже файл с похожей темой? → grep по названию
2. Это временная заметка? → не создавай отдельный файл
3. Это статус проекта? → обновляй существующий документ (README / ROADMAP / PROJECT.md)
4. Это дорожная карта? → дополни существующий роадмап
5. Только если НИ ОДИН файл не подходит → создавай новый

## Запрещённые паттерны
| ❌ Запрещено | ✅ Правильно |
|-------------|-------------|
| `*_V2.md`, `*_NEW.md` | Обновить оригинал |
| `TODO.md`, `STATUS.md` | Основной документ проекта |
| `PLAN_*.md` | ROADMAP проекта |
| Временные файлы | Удалить после сессии |

## Структура проекта (ИДЕАЛ)
```
любой_проект/
├── .opencode/         ← Конфигурация opencode
│   ├── opencode.json
│   └── skills/        ← Проектные скиллы
├── AGENTS.md          ← Инструкции для AI-агента (читать при старте)
├── README.md          ← Что за проект
├── ROADMAP.md         ← Дорожная карта (единый файл!)
├── CHRONICLE.md       ← Хронология (опционально)
├── src/               ← Исходный код
└── [специфичные файлы проекта]
```

## Правила общения агента
- Общаться на русском языке
- Не повторять информацию, которую пользователь уже знает
- Не предлагать пользователю сделать то, что можно выполнить самому
- Быть кратким, по делу
- НЕ создавать документацию без явного запроса

## Команды для opencode
- `/simple` — Tier 1 (DeepSeek Flash Free, простые задачи)
- `/standard` — Tier 2 (DeepSeek V4 Pro, по умолчанию)
- `/advanced` — Tier 3 (MiniMax M2, архитектура)
- `/critical` — Tier 4 (Claude Opus 4, продакшен)

## Progressive Disclosure
НЕ читай все скиллы сразу — загружай по мере необходимости:
1. При старте: AGENTS.md + CHRONICLE.md (если есть)
2. При задаче: найди релевантный скилл → прочитай → следуй
3. После задачи: контекст скилла не нужен

## Context Engineering — управление контекстом агента

### Иерархия контекста (от постоянного к временному)
```
1. Rules / AGENTS.md           ← Всегда загружено, проект-wide
2. Spec / Architecture Docs    ← Загружать под задачу
3. Relevant Source Files       ← Загружать под задачу
4. Error Output / Test Results ← Загружать под итерацию
5. Conversation History        ← Накапливается, компактить
```

### Правила загрузки контекста
- **Контекстное голодание:** агент выдумывает API, игнорирует конвенции. Решение: загружать rules + релевантные файлы под каждую задачу.
- **Контекстное наводнение:** больше файлов ≠ лучше результат. Цель: < 2000 строк focused контекста на задачу.
- **Устаревший контекст:** агент ссылается на удалённый код. Решение: свежая сессия при дрифте.
- **Отсутствие примеров:** агент изобретает новый стиль. Решение: показать 1 пример паттерна из кодовой базы.
- **Неявное знание:** если не записано — его не существует. Всё в файлы.

### Когда контекст конфликтует
Не выбирай молча. Поверхностно:
```
КОНФЛИКТ:
Спецификация говорит X, но код использует Y
→ Какой подход выбрать?
```

### Inline Planning Pattern
Для многошаговых задач — эмитируй лёгкий план перед выполнением:
```
PLAN:
1. Сделать A
2. Сделать B
3. Сделать C
→ Выполняю, если не перенаправите.
```
Это 30 секунд, которые предотвращают 30 минут переделок.
