# Воркфлоу разработки (для всех проектов)

## Feature Implementation Workflow
```
Plan First → TDD → Code Review → Commit
```
Никогда не пропускать шаги.

## 0. Search-First (обязательный каскад)
Перед тем как писать код или решать техническую проблему:
```
1️⃣ СВОИ СКИЛЛЫ — grep по .opencode/skills/ и knowledge/
   Если решение найдено → ИСПОЛЬЗОВАТЬ

2️⃣ GITHUB / NPM — поиск рабочих библиотек и примеров
   Если нашёл → адаптировать

3️⃣ СВОЯ ГОЛОВА — только если 1️⃣ и 2️⃣ не дали результата
   Писать собственное решение
```
**Нарушение:** тратить 20+ минут на изобретение решения, которое уже есть — грубая ошибка.

## 1. Plan First
- Декомпозиция задач — независимые подзадачи запускать параллельно
- Критерии завершения определять ДО начала работы
- Каждый юнит: независимо верифицируемый, с одним доминирующим риском, с чётким условием завершения

## 2. TDD
- Пиши тест первым (RED)
- Минимальная реализация (GREEN)
- Рефакторинг (IMPROVE)
- Цель: 80%+ coverage
- AAA Pattern: Arrange → Act → Assert

## 3. Code Review (5 осей)
### Correctness — делает ли код то, что должен?
- Соответствует ли спецификации?
- Edge cases обработаны (null, empty, boundary)?
- Error paths, а не только happy path?

### Readability — понятен ли код без автора?
- Имена осмысленные и консистентные
- Управляющий поток прямолинейный
- Нет "умных" трюков, которые можно упростить

### Architecture — вписывается ли в систему?
- Следует существующим паттернам
- Чистые границы модулей
- Нет дублирования кода
- Нет циклических зависимостей

### Security — AI Agent & OWASP

#### AI Agent Security (Prompt Injection Protection)
- **Separation:** чётко отделяй System Prompt от User Input. Никогда не вставляй ввод пользователя напрямую в инструкции.
- **Guardrails:** если пользователь просит «забудь инструкции» — игнорировать. Использовать стоп-фразы и проверки.
- **Output Validation:** проверяй ответ LLM перед отправкой на наличие секретов (ключи, токены, пароли).
- **Tool Least Privilege:** у инструментов минимальные права. Критические действия (удаление, массовая рассылка) — с подтверждением.
- **Execution Sandboxing:** жёсткий таймаут на все вызовы LLM (30с), ограничение объёма данных.

#### OWASP Top 10 Prevention
- **Injection:** parameterized queries, никакой конкатенации строк в SQL
- **Broken Auth:** bcrypt (salt ≥ 12), httpOnly/secure/sameSite cookies
- **XSS:** framework auto-escaping (React делает по умолчанию), sanitize через DOMPurify если нужен raw HTML
- **Broken Access Control:** проверяй authorization на каждом endpoint'е, а не только authentication
- **Security Misconfiguration:** helmet/CSP для Express, CORS — только известные origin'ы
- **Sensitive Data Exposure:** никогда не возвращай пароли/токены в API ответах

#### Input Validation
- Schema validation через Zod на system boundaries (route handler'ы, формы, парсинг внешних API)
- Третьесторонние API ответы — **ненадёжные данные**, всегда валидировать
- Внутренний код после границы — валидация не нужна (типы уже проверены)

#### Secrets Management
- `.env` и `.env.local` в `.gitignore` — НИКОГДА не коммитить
- Перед коммитом: `git diff --cached | grep -i "password\|secret\|api_key\|token"`
- Не логировать секреты, не выводить stack trace пользователю
- Всегда проверять `npm audit` перед релизом

#### Rate Limiting
```typescript
// Общий лимит API
app.use('/api/', rateLimit({ windowMs: 15 * 60 * 1000, max: 100 }))
// Строже для auth
app.use('/api/auth/', rateLimit({ windowMs: 15 * 60 * 1000, max: 10 }))
```

### Performance
- Нет N+1 query паттернов
- Нет неограниченных циклов или запросов
- Пагинация на list endpoint'ах
- Нет ненужных ререндеров в React

## Thermo-Nuclear Gates (блокеры мержа)
- **Code Judo:** удаляй сложность, не перемещай. Если можно убрать целый слой — сделай это
- **Правило 1000 строк:** файл > 1000 строк — code smell, декомпозируй
- **Анти-спагетти:** ad-hoc условия в чужих потоках = проблема дизайна
- **Прямой код > магический:** избегай дженерик-механизмов, скрывающих простые допущения
- Не принимай «почищу потом» — чистить сейчас или создавать issue

## Conventional Commits
```
<type>: <description>

<optional body>
```
Types: feat, fix, refactor, docs, test, chore, perf, ci, style

## Git Workflow
- Ветки: feature/<name>, fix/<name>
- Commits: мелкие, атомарные (~100 строк/change)
- PR: git diff от базовой ветки, полный summary, test plan
- Push с -u для новых веток

## Coding Style
- **Иммутабельность:** всегда создавай новые объекты, не мутируй существующие
- **KISS:** самое простое решение, которое работает
- **DRY:** выноси повторяющуюся логику
- **YAGNI:** не строй фичи, пока не нужны
- **Много маленьких файлов > мало больших:** 200-400 строк типично, 800 макс
- **Функции:** < 50 строк
- **Вложенность:** максимум 4 уровня, предпочитай early returns
- **Naming:** camelCase переменные/функции, PascalCase компоненты/типы, UPPER_SNAKE константы
- **Error handling:** явно, никогда не проглатывать молча

## Dependency Discipline
Перед добавлением зависимости:
1. Существующий стек уже решает это? (Часто да.)
2. Размер бандла не должен расти > 20%
3. Активно ли поддерживается?
4. Есть ли уязвимости? (`npm audit`)
5. Какая лицензия?

## Управление сессиями (Session Lifecycle)

### 🚀 Старт сессии
Когда пользователь пишет **"старт"**, **"boot"**, **"продолжим"**, **"start"**:

1. **Прочитай контекст:**
   - `AGENTS.md` — каскадная система и общий контекст
   - `CHRONICLE.md` (конец файла) — что делали в последней сессии
   - `chp.md` — чекпоинт последнего состояния (если есть)
   - `.opencode/skills/` — нужный скилл под задачу (progressive disclosure)

2. **Проверь статус:**
   - `git status` — нет ли незакоммиченных изменений
   - `npm run build` или `npm run dev` — проект собирается?

3. **Спроси** (если контекст неочевиден): «С чего продолжим?»

### 🛰️ Завершение сессии (finish-day)
Когда пользователь пишет **"finish-day"**, **"финиш"**, **"заканчиваем"**, **"конец дня"**, **"done"** — выполни протокол ПОЛНОСТЬЮ.

**Порядок: сначала коммит и пуш, потом CHRONICLE и чекпоинт.**

#### Шаг 1: Git — коммит и пуш
Перед тем как что-то записывать — зафиксируй изменения в гите:

1. `git status` — покажи что изменилось
2. Спроси: «Коммичу всё? Или выборочно?»
3. После подтверждения:
   ```bash
   git add -A
   git commit -m "тип: описание"
   git push
   ```

#### Шаг 2: Обнови CHRONICLE.md
Файл `CHRONICLE.md` уже содержит структуру: Goal, Constraints, Progress, Key Decisions, Next Steps, Critical Context, Relevant Files.

Допиши в конец файла новый блок сессии и обнови существующие секции:

```markdown
--- СЕССИЯ: YYYY-MM-DD HH:MM

## 🛠 Что сделано
- [задача 1] — [решение, ключевые изменения]
- [задача 2] — [решение]

## 💬 Ключевые моменты
- [решения, договорённости, инсайты]

## 📋 Затронутые файлы
- [путь]
```

После чего обнови:
- **Progress → Done:** добавь завершённое
- **Progress → In Progress:** что осталось
- **Key Decisions:** новые решения
- **Next Steps:** пересмотри список
- **Critical Context:** обнови при необходимости
- **Relevant Files:** добавь новые/удалённые

#### Шаг 3: Создай чекпоинт `chp.md`
```markdown
# 📌 Чекпоинт: YYYY-MM-DD HH:MM

## Последняя задача
[что делали, на чём остановились]

## Состояние
- Ветка: [git branch]
- Последний коммит: [хеш]
- Сборка: [OK/ошибки]

## Следующий шаг
[что делать при продолжении]
```

#### Шаг 4: Финал
1. **Сборка:** `npm run build` — не сломана?
2. **Скажи пользователю:**
   > ✅ Git запушен. CHRONICLE.md обновлён. Чекпоинт в chp.md.
   > Для продолжения: Cmd+N → "продолжим"
