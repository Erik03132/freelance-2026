# Freelance Agent MVP — Спецификация проекта

## 1. Цель и общая концепция

Создать автономное приложение‑агент, который:

- По расписанию (например, каждый день в 09:00) заходит на выбранные фриланс‑биржи (через браузерный интерфейс, без официальных API).
- Находит новые задания, автоматически сопоставляет их с моим профилем навыков и отбирает только подходящие (порог совпадения навыков ≥ 70%).
- Отбирает задания с достаточно подробным и понятным ТЗ.
- Для выбранных задач готовит предпродажный материал: прототип, демо, черновой макет, фрагмент кода и т.п.
- Формирует безопасный превью‑результат (водяные знаки, частичная функциональность, деплой на мой хостинг и т.д.), чтобы клиент не мог бесплатно утащить результат.
- Уведомляет меня о подходящих задачах и, по желанию, генерирует черновик предложения/ответа клиенту.

---

## 2. Целевые площадки и юридические ограничения

Агент должен учитывать, что у крупных бирж есть строгие правила по скрейпингу и автоматизации (Upwork, Fiverr, Freelancer и т.д.).

**Входные данные (параметры):**
- Список площадок: URL страниц с лентами задач (поиск, категории, фильтры).
- Формат авторизации: логин/пароль, 2FA, куки (вводятся пользователем вручную).

**Требования по соблюдению правил:**
- Использовать только эмуляцию браузера (Antigravity‑агент в браузере): прокликивание, чтение DOM, скролл, без прямых «скрейперных» запросов в обход интерфейса.
- Настроить «человеческие» задержки, ограничение по количеству запросов (чтобы не вызвать блокировку).
- Не обходить paywall, капчи и технические средства защиты.
- В настройках предусмотреть флажок «Я принимаю риск и ответственность за автоматизацию на этих сайтах» — агент не принимает решения за пользователя.

---

## 3. Архитектура приложения и режимы работы

Приложение должно быть построено вокруг агентного подхода Antigravity, где агенты могут взаимодействовать с браузером, редактором и терминалом.

**Минимальные модули:**

### 1) Scheduler‑агент (Планировщик)
- Запускается по расписанию (по умолчанию 09:00 каждый день).
- Стартует цепочку: авторизация → обход площадок → парсинг → фильтрация → генерация предпродажных материалов.

### 2) Scraper‑агент (Браузерный парсер)
- Работает внутри браузера: логинится, листает страницы заданий, открывает карточки проектов, извлекает текстовые описания, бюджет, сроки, теги навыков.
- Обрабатывает несколько площадок последовательно, но с общим интерфейсом (каждая площадка — отдельный сценарий действий).

### 3) Matching‑агент (Оценка соответствия навыков)
- Сравнивает описание заданий с моим профилем навыков.
- Вычисляет «процент совпадения» на основе: ключевых слов, семантической близости, категорий навыков.
- Отбирает только те задачи, где score ≥ заданного порога (например, 0.7 по шкале 0–1).

### 4) Filter‑агент по качеству ТЗ
- Оценивает «полноту/четкость ТЗ» по эвристикам:
  - Есть ли описание цели проекта.
  - Есть ли технические детали (стек, платформы, интеграции, примеры).
  - Есть ли входные данные/материалы (макеты, примеры, ссылки).
- Присваивает заданиям «оценку понятности ТЗ» (0–1) и отбрасывает задачи ниже порога (например, 0.6).

### 5) Solution‑Draft‑агент (Предварительное решение)
- Для каждой отфильтрованной задачи:
  - Генерирует план решения (структурированное техрешение).
  - При необходимости создает прототип:
    - Если задача про сайт — scaffold проекта (например, Next.js/React), базовую верстку ключевых страниц, моковые данные.
    - Если задача про дизайн — набор превью‑изображений с водяными знаками.
  - Подготавливает «ограниченную версию» результата, пригодную для демонстрации, но не для полного использования.

### 6) Protection‑агент (Страховочные меры)
- Реализует меры защиты от недобросовестных заказчиков:
  - Веб‑проекты деплоятся на мой хостинг/сервер с ограниченным функционалом (например, отсутствует export, ограничение по роутам, нет исходников).
  - Дизайн и графика — с водяными знаками (лого, email, текст).
  - Код — только фрагменты/snippet'ы, без полного решения критичных частей.
- Готовит список «что будет убрано/открыто только после оплаты» (чтобы можно было ясно написать клиенту).

### 7) Proposal‑агент (Подготовка письма/ответа)
- Формирует черновик сообщения клиенту:
  - Кратко пересказывает задачу своими словами.
  - Выписывает, почему я подхожу (навыки, опыт).
  - Прикладывает ссылку на демо/прототип.
  - Обозначает формат оплаты и следующие шаги.
- Реализовать так, чтобы я мог вручную редактировать/утверждать письмо перед отправкой.

### 8) UI‑панель (Dashboard)
- Список найденных сегодня задач: платформа, заголовок, бюджет, срок, процент совпадения, понятность ТЗ.
- Статусы: «новое», «прототип готов», «отправлено предложение», «в процессе переговоров», «закрыто».
- Настройки (см. следующий раздел).

## 4. Настройки профиля и навыков

Нужен отдельный блок «Профиль и навыки»:

**Основная информация:** имя, ник, ссылки на портфолио (GitHub, сайт, Dribbble и т.п.).

**Список навыков с уровнями:**
- Пример: `React – 0.9, Next.js – 0.8, Node.js – 0.7, UI/UX – 0.6` (шкала 0–1).

**Категории задач, которые мне интересны:**
- Web‑разработка, бекенд, интеграции, дизайн, лендинги, мобильные приложения и т.д.

**Типы задач, которые нужно игнорировать:**
- Написание текстов, SMM, перевод, «серые/сомнительные» темы (задания с ключевыми словами: «ботнет», «кардинг», «спам» и т.п.).

**Настройки сопоставления:**
- Порог матч‑скора (по умолчанию 0.7).
- Вес навыков против опыта (если будет реализован импорт резюме/портфолио).
- Возможность ручного исключения конкретных заказчиков/ключевых слов.

## 5. Логика матчинга задач и ТЗ

### 5.1 Матчинг навыков

Использовать комбинированный подход:
- Словарный/ключевой (точное совпадение навыков и стеков).
- Семантическая близость (синонимы, похожие фреймворки).

**Пример логики:**
- Выделить из описания задачи набор skills‑tokens.
- Сопоставить их с моим списком навыков.
- Для каждого совпавшего навыка суммировать веса, нормировать на максимум.
- Получить итоговый score ∈ [0, 1].

### 5.2 Оценка понятности/качества ТЗ

Ввести простую «скоринговую модель»:

**Признаки, повышающие score:**
- Указан стек технологий.
- Есть примеры/референсы.
- Описаны пользовательские сценарии.
- Указаны формальные критерии готовности (acceptance criteria).

**Признаки, понижающие score:**
- «Сделать сайт как‑нибудь», «сделать красиво», «подумаем по ходу».
- Мало деталей, только 1–2 предложения.

**Результат:** `clarity_score` ∈ [0, 1]; заявки ниже порога не отправляем в автогенерацию прототипа.

## 6. Механизмы защиты от недобросовестных заказчиков

**Общие принципы:**
- Демо должно показать компетенцию, но не давать рабочее «боевое» решение.
- То, что легко скопировать, должно быть урезано или защищено.

**Конкретные меры:**

### 1) Веб‑проекты
- Деплой только на мой домен/хостинг.
- Без доступа клиента к репозиторию/архиву проекта.
- Критичные части (уникальные алгоритмы, интеграции, инфраструктура) — не реализуются в демо или реализуются в виде моков.

### 2) Дизайн/графика
- Автоматическое наложение водяных знаков (логотип, url, email) на все превью‑картинки.
- Низкое разрешение превью (например, 1200 px по длинной стороне).

### 3) Код/скрипты
- Демонстрировать структуру и ключевые идеи, но не полный production‑код.
- Ограничиться фрагментами в сообщении плюс ссылка на демонстрацию.

### 4) Юридические оговорки
- Типовой текст в сообщении:
  - «Данное демо предоставлено исключительно в ознакомительных целях, права на решение и исходный код передаются только после оплаты и заключения договора».

---

## 7. Управление рисками и анти‑спам логика

Чтобы не «перегорать» клиентов и не попасть в бан на платформах:

- **Ограничения по количеству задач в день:**
  - Максимум X новых задач для генерации прототипа в сутки (X настраивается).
- **Ограничение по авто‑отправке:**
  - По умолчанию агент только готовит черновики и ждет моего подтверждения.
- **Шаблоны сообщений** должны быть персонализированы, с отсылкой к конкретным деталям ТЗ, чтобы не выглядеть спамом.
- **Встроенный список стоп‑слов** и стоп‑фраз для задач, которые нужно игнорировать по этическим или юридическим причинам.

## 8. Интеграция с Antigravity и технические требования

С учетом того, что Antigravity — IDE и платформа для агентного программирования с доступом к браузеру, редактору и терминалу:

**Агент должен уметь:**
- Управлять браузером: открывать вкладки, логиниться, кликать, читать HTML/DOM, скроллить.
- Создавать/редактировать файлы проекта в локальном workspace (через IDE‑интерфейс).
- Запускать команды деплоя (например, `npm install`, `npm run build`, `npm run deploy` или команды для Docker), если этому явно разрешено.

**Guardrails (защитные механизмы):**
- Логи всех действий агента.
- Возможность стопнуть/приостановить агента.
- Настройка прав: каким сайтам можно давать доступ, какие команды в терминале разрешены.

## 9. Минимальный MVP и приоритеты

**Очередность разработки (для агента):**

### MVP №1 (Минимально работающий)
- Одну‑две конкретные площадки.
- Логин, сбор заданий за последние 24 часа.
- Простая фильтрация по ключевым словам и ручному списку навыков.
- Список задач + черновики писем без прототипов.

### MVP №2
- Внедрение семантического матчинга навыков (score ≥ 0.7).
- Оценка понятности ТЗ.
- Автогенерация текстового плана решения и базового прототипа (например, шаблон сайта).

### MVP №3
- Меры защиты (водяные знаки, деплой демо на свой хостинг).
- Расширение на несколько площадок.
- Нормальный дашборд с фильтрами и статусами задач.

## 10. Дополнительные пожелания к функционалу

- **Возможность «дообучить» матчер** на моих прошлых проектах:
  - Загрузить список задач, которые я успешно сделал, и тех, которые не подходят, чтобы улучшить алгоритм.
- **Отчеты:**
  - Еженедельная статистика — сколько задач найдено, сколько прототипов сделано, сколько ответов отправлено, сколько диалогов началось.
- **Режим «ручной проверки»:**
  - Возможность вручную пометить задания как «интересно/не интересно», чтобы система подстраивалась.
